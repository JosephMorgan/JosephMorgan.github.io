JMeter的基础概念

测试计划元件

Test Plan(测试计划)：用来描述一个测试，包含与本次测试所有相关的功能，作用于整个测试。

Threads（Users）线程用户：有三个添加线程组的选项，名字不一样， 创建后，其界面是完全一样的。

  • setup thread group：一种特殊类型的ThreadGroup，用于执行测试前的操作。
  • teardown thread group：一种特殊类型的ThreadGroup，用于执行测试后的操作。
  • thread group(线程组)：一般用到的线程组，一个线程组,，可以看做一个虚拟用户组，线程组中的每个线程都可以理解为一个虚拟用户。
  • Test Fragmen（测试片段）：一种特殊的线程组，它在测试树上与线程组处于一个层级。它与线程组有所不同，因为它不被执行，除非它是一个模块控制器或者是被控制器所引用时才会被执行。

控制器，
JMeter有两种类型的控制器：采样器（Sampler）和逻辑控制器（Logic Controller），这些元件是用来驱动处理一个测试。

Sampler：采样器（Sampler）是测试中向服务器发送请求，记录响应信息，记录响应时间的最小单元，JMeter 原生支持多种不同的Sampler ，如 HTTP Request Sampler 、 FTP Request Sampler 、TCP Request Sampler 、JDBC Request Sampler等，每一种不同类型的Sampler可以根据设置的参数向服务器发出不同类型的请求。另外还有Java Request Sampler 和 Beanshell Request Sampler。

逻辑控制器：有两类元件，一类是控制测试计划执行过程中节点的逻辑执行顺序，常用的有，如果（If）控制器 、switch Controller 、Runtime Controller、循环控制器等。另一类是对测试计划中的脚本进行分组、方便JMeter统计执行结果以及进行脚本的运行时控制，如事务控制器、吞吐量控制器等。

配置元件：配置元件（config element）用于提供对静态数据配置的支持。CSV Data Set config 可以将本地数据文件形成数据池（Data Pool），而对应于HTTP Request Sampler和 TCP Request Sampler等类型的配置元件则可以修改Sampler的默认数据。如，HTTP Cookie Manager 可以用于对HTTP Request Sampler的cookie进行管理。

定时器：定时器（Timer）用于操作之间设置等待时间，等待时间是测试中常用的控制客户端QPS的手端。类似于LoadRunner里面的“思考时间”。JMeter 定义了Bean Shell Timer、Constant Throughput Timer、固定定时器等不同类型的Timer。

前置处理器：用于在实际的请求发出之前对即将发出的请求进行特殊处理。例如，HTTP URL重写修复符则可以实现URL重写，当URL中有sessionID 一类的session信息时，可以通过该处理器填写发出请求的实际的sessionID 。

后置处理器：用于对Sampler发出请求后得到的服务器响应进行处理。一般用来提取响应中的特定数据（类似LoadRunner测试工具中的关联概念）。例如，XPath Extractor则可以用于提取响应数据中通过给定XPath值获得的数据。

断言：用于检查测试中得到的相应数据等是否符合预期，断言一般用来设置检查点，用以保证测试过程中的数据交互是否与预期一致。

监听器：用来对测试结果数据进行处理和可视化展示的一系列元件。 图形结果、察看结果树、聚合报告。都是经常会用到的元件。

线程组中

线程数：虚拟用户数。一个虚拟用户占用一个进程或线程。设置多少虚拟用户数在这里也就是设置多少个线程数。

Ramp-Up Period(in seconds)：准备时间，设置的虚拟用户数需要多长时间全部启动。如果线程数为20 ，准备时间为10 ，那么需要10秒钟启动20个线程。也就是每秒启动2个线程。若要立刻全部启动，则设为0。

循环次数：每个线程发送请求的次数。如果线程数为20 ，循环次数为100 ，那么每个线程发送100次请求。总请求数为20*100=2000 。如果勾选了“永远”，那么所有线程会一直发送请求，直到选择停止运行脚本。

另外，还可以灵活的选择设定测试运行时间。勾选“调度器”，进行调度器配置。

发送json数据

1.用CSV Data set Config,进行参数化。

2.直接在请求Sampler当中，把json字符串写在Post　Body。

3.直接写在Parameters中。

注意：发送json格式的请求的时候，需要在信息头管理器设置Content-Type=application/json 。

插件下载

下载地址：https://jmeter-plugins.org/ 。另外，插件管理还有 Plugins Manager。

用于服务器性能监视的PerfMon Metrics Collector
用于建立压力变化模型的Stepping Thread Group
用于Json解析的JSON Path Extractor
用于展示响应时间曲线的Response Times Over Time
用于展示TPS曲线的Transactions per Second

集合点：请求—–>定时器——->Synchronizing Timer，“Number of Simulated Users to Group by: 5”，表示并发量是5。也就是说，JMeter会等到5个线程都运行到此处时，这5个线程才开始执行后面的操作，先到达的线程将会在此等候。注意，这个值不能大于该线程组的线程数。

关联

JMeter的关联方法有两种：后置处理器－正则表达式提取器与XPath Extractor。

引用名称：下一个请求要引用的参数名称，如填写activityID，则可用${activityID}引用它。

正则表达式：()括起来的部分就是要提取的。 .匹配任何字符串。*：零次或多次。+：一次或多次。?：不要太贪婪，零次或一次，在找到第一个匹配项后停止。

模板：用$$引用起来，如果在正则表达式中有多个正则表达式（多个括号括起来的东东），则可以是$2$$3$等等，表示解析到的第几个值给title。如：$1$表示解析到的第1个值

匹配数字：0代表随机取值，1代表全部取值，通常情况下填0，如果在LR中，取出的值是一个数组，还得处理一下，LR11版本用一个随机的函数就可以不用写大段的代码来处理数组。

缺省值：如果参数没有取得到值，那默认给一个值让它取。

XPath Extractor的设置

Use Tidy：当需要处理的页面是HTML格式时，必须选中该选项，当需要处理的页面是XML或XHTML格式（例如，RSS返回）时，取消选中该选项。

Reference Name：存放提取出的值的参数。

XPath Query：用于提取值的XPath表达式。

Default Value：参数的默认值。
